# OCI NixOS マイグレーション 実装計画

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** 4サービス (adguard-home, vaultwarden, gatus, backup-kopia-b2) を OCI Always Free (Ampere A1 aarch64) の NixOS に移行する

**Architecture:** nixos-anywhere で Ubuntu ARM VM を NixOS 化し、`virtualisation.oci-containers` + sops-nix でサービスを宣言的に管理する。既存の Tailscale sidecar パターンを維持しつつ、Quadlet テンプレート + gomplate を NixOS モジュールに置き換える。

**Tech Stack:** NixOS, disko, sops-nix, Podman (rootful), Tailscale, SOPS + age

**Spec:** `docs/superpowers/specs/2026-04-14-oci-nixos-migration-design.md`

```nix
        group: services

  # --- Secrets ---
  sops.secrets."gatus/ts_authkey" = {
    sopsFile = ./secrets.yaml;
    restartUnits = [ "podman-gatus-ts.service" ];
  };
  sops.secrets."gatus/telegram_bot_token" = {
    sopsFile = ./secrets.yaml;
    key = "TELEGRAM_BOT_TOKEN";
    restartUnits = [ "podman-gatus.service" ];
  };
  sops.secrets."gatus/telegram_chat_id" = {
    sopsFile = ./secrets.yaml;
    key = "TELEGRAM_CHAT_ID";
    restartUnits = [ "podman-gatus.service" ];
  };

  # --- Tailscale sidecar ---
  virtualisation.oci-containers.containers.gatus-ts = {
    image = "docker.io/tailscale/tailscale:latest";
    autoStart = true;
    environment = {
      TS_HOSTNAME = "gatus";
      TS_STATE_DIR = "/var/lib/tailscale";
      TS_SERVE_CONFIG = "/config/serve.json";
      TS_USERSPACE = "true";
    };
    environmentFiles = [
      config.sops.secrets."gatus/ts_authkey".path
    ];
    volumes = [
      "gatus-ts-state:/var/lib/tailscale"
      "${pkgs.writeText "gatus-ts-serve.json" tsServeConfig}:/config/serve.json:ro"
    ];
    extraOptions = [
      "--health-cmd=tailscale status --json | grep -q '\"Online\": true' || exit 1"
      "--health-interval=30s"
      "--health-timeout=10s"
      "--health-retries=3"
      "--health-start-period=60s"
    ];
  };

  # --- Gatus ---
  virtualisation.oci-containers.containers.gatus = {
    image = "ghcr.io/twin/gatus:stable";
    autoStart = true;
    dependsOn = [ "gatus-ts" ];
    environmentFiles = [
      config.sops.secrets."gatus/telegram_bot_token".path
      config.sops.secrets."gatus/telegram_chat_id".path
    ];
    extraOptions = [
      "--network=container:gatus-ts"
    ];
    volumes = [
      "gatus-data:/data"
      "${gatusConfig}:/config/config.yaml:ro"
    ];
  };
}
```

Note: Gatus のエンドポイント URL を Tailscale の MagicDNS ホスト名 (`vaultwarden:443`, `adguard-home:443`) に変更している。OCI 上では全サービスが Tailnet 経由で通信するため、Tailscale Serve の FQDN ではなくショートネームで到達可能。
Gatus config 内の `${VAR}` は Gatus 自体の環境変数展開機能を利用する。Nix の文字列内では `''${VAR}` でエスケープする。

Telegram シークレットの sops 格納形式について: 既存の secrets.yaml では `TELEGRAM_BOT_TOKEN` と `TELEGRAM_CHAT_ID` がトップレベルキーとして格納されている。sops-nix の `key` パラメータでこれを参照する。environmentFiles にはこれらが `KEY=VALUE` 形式のファイルとして渡される。

**重要:** sops-nix の `environmentFiles` に渡すシークレットは、ファイル内容が `KEY=VALUE` 形式である必要がある。sops-nix はデフォルトでは値のみをファイルに書き出す。secrets.yaml のキー名がそのまま環境変数名になるよう、sops-nix の `path` で出力されるファイルを使い、コンテナ起動時にスクリプトで環境変数を組み立てる方式が必要になる可能性がある。この点は実装時に検証し、必要であれば systemd の `ExecStartPre` で env ファイルを生成するアプローチに切り替える。

backup-kopia-b2 は他のサービスと異なり、コンテナではなく直接 kopia CLI を systemd で実行する。
NixOS では systemd サービス + タイマーとして定義する。

```nix
# services/backup-kopia-b2/nixos.nix
{ config, pkgs, ... }:
{
  # --- Secrets ---
  sops.secrets."backup/b2_endpoint" = {
    sopsFile = ./secrets.yaml;
    key = "b2_endpoint";
  };
  sops.secrets."backup/b2_bucket" = {
    sopsFile = ./secrets.yaml;
    key = "b2_bucket";
  };
  sops.secrets."backup/b2_key_id" = {
    sopsFile = ./secrets.yaml;
    key = "b2_key_id";
  };
  sops.secrets."backup/b2_secret_key" = {
    sopsFile = ./secrets.yaml;
    key = "b2_secret_key";
  };
  sops.secrets."backup/kopia_password" = {
    sopsFile = ./secrets.yaml;
    key = "kopia_password";
  };

  # --- Packages ---
  environment.systemPackages = [ pkgs.kopia ];

  # --- Kopia backup service (oneshot) ---
  systemd.services.kopia-backup = {
    description = "Kopia snapshot backup";
    after = [ "network-online.target" ];
    wants = [ "network-online.target" ];
    serviceConfig = {
      Type = "oneshot";
      ExecStart = "${pkgs.kopia}/bin/kopia --config-file=/var/lib/kopia/vault-b2.config snapshot create --all";
    };
  };

  # --- Daily timer ---
  systemd.timers.kopia-backup = {
    description = "Daily Kopia backup";
    wantedBy = [ "timers.target" ];
    timerConfig = {
      OnCalendar = "daily";
      RandomizedDelaySec = "1h";
      Persistent = true;
    };
  };
}
```

Note: バックアップソースが元マシンにあるため、このサービスは OCI 上ではスタブ状態。
`kopia repository connect` と `kopia snapshot create --all` のソース設定は、バックアップ対象を OCI に移動/マウントした後に手動で行う。
config ファイルのパスを `/var/lib/kopia/vault-b2.config` に変更している (rootful systemd のため `~` が `/root` になる問題を回避)。
